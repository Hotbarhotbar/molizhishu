# 五阶段演进路线图

从人机协作到全AI自动化的渐进式演进计划。

---

> **Coze 替代讨论**：详见 [06-coze-replacement-analysis.md](06-coze-replacement-analysis.md)。推荐 Phase 1 用 Claude Code 原生（CronCreate + Agent + WebSearch + 飞书 Webhook）替代 Coze，开发量更小且无平台依赖。

## Phase 1：选题AI化（当前阶段）

**时间**：立即启动
**目标**：选题环节从"人工刷热点"变成"AI推送、人工确认"

```
Coze 扫描7平台热点
  → 知识库规则评估+三维打分
  → 生成19字段选题Brief
  → 飞书卡片推送到群聊
  → 🧑 人工看卡片，选今天写哪个
  → Claude Code 写稿
  → 🧑 人工修改+发布
```

**Coze 端建设内容**：
- 选题 Bot（含扫描→评估→生成Brief 工作流）
- 知识库（同步 AI_Writing_Vault 核心规则）
- 飞书机器人插件（卡片消息推送）

**人工保留节点**：选题决策、文章修改、发布确认

---

## Phase 2：AI辅助评审

**时间**：Phase 1 稳定运行2-4周后
**目标**：Coze 评审Agent 对选题给出首次评审意见，人工参考AI意见做决策

```
Coze 扫描 → 评估 → 生成Brief
  → Coze 评审Agent 首次打分+通过/驳回建议
  → 飞书推送（选题卡片 + 🤖 AI评审意见）
  → 🧑 参考AI意见，确认选题
  → Claude Code 写稿
  → 🧑 人工修改+发布
```

**新增能力**：
- 评审Agent（对比历史已发文章做去重检查、结构相似性检查、风险预判）
- 飞书卡片增加"AI评审意见"区域

**人工保留节点**：最终选题决策、文章修改、发布确认

---

## Phase 3：AI自主评审

**时间**：Phase 2 积累足够评审数据后
**目标**：AI自主决定选题通过/驳回，人工仅接收通知

```
Coze 扫描 → 评估 → 生成Brief → 评审Agent 自主通过/驳回
  → 通过的选题自动进入写作队列
  → 飞书推送通知（仅通知，不阻塞流程）
  → Claude Code 自动拉取Brief写初稿
  → 🧑 人工修改+发布
```

**新增能力**：
- 评审Agent 获得自主决策权（高分自动通过、低分自动驳回、边界case人工）
- Claude Code 支持自动拉取 Brief 启动写作

**审批模式切换**：
- 选题分 ≥ 13：自动通过，飞书静默通知
- 选题分 10-12：飞书推送建议通过，24h无人驳回即通过
- 选题分 < 10：自动驳回，不推送
- 人工可随时介入驳回任何自动通过的选题

**人工保留节点**：文章修改、发布确认

---

## Phase 4：全流程AI（写作→修改→评分自动化）

**时间**：Phase 3 写作质量和风格稳定后
**目标**：写作、修改、评分全由AI Worker完成，人工仅做发布确认

```
Coze 总控：
  选题Agent → 评审Agent → 写作Agent → 修改Agent → 评分Agent
  → 飞书推送终稿（含AI评分+修改记录）
  → 🧑 人工确认发布
```

**新增 Worker Agent**：

| Agent | 职责 | 依赖 |
|-------|------|------|
| 写作Agent | 根据Brief+风格卡生成初稿（4种写作模式可选） | AI_Writing_Vault 写作规则 |
| 修改Agent | 9项反AI检测清单检查、句长抖动、AI词替换 | 品牌声音文档 |
| 评分Agent | 5维度自检打分（必须<35分合格）、质量门禁 | 发布前检查清单 |

**质量门禁**：
- 评分Agent 打分 < 35 → 自动进入发布队列
- 评分Agent 打分 35-50 → 退回修改Agent重写（最多3次）
- 评分Agent 打分 > 50 → 退回写作Agent重写
- 3次修改仍不合格 → 飞书告警，人工介入

**人工保留节点**：发布确认

---

## Phase 5：AI客户经理制（终局形态）

**时间**：Phase 4 稳定运行、质量可靠后
**目标**："客户经理Agent"统筹全流程，"工人Agent"执行具体任务，人工退到纯监督位

```
┌──────────────────────────────────────────────┐
│             Coze 总控层                       │
│                                               │
│  ┌─────────────────────────────────────┐     │
│  │      客户经理 Agent（Manager）       │     │
│  │                                      │     │
│  │  懂上下文（知道历史发过什么）          │     │
│  │  会判断（选题不行主动建议不发）        │     │
│  │  能兜底（工人出问题调动补救）          │     │
│  │                                      │     │
│  │  每日播报："老板，今天3个选题≥12分，  │     │
│  │   我选了最热的豆包2.1，写完了，改2稿， │     │
│  │   评分85，发到草稿箱了，你看看"        │     │
│  └──────────┬──────────────────────────┘     │
│             │                                 │
│  ┌──────────┼──────────┬──────────┬──────┐   │
│  ↓          ↓          ↓          ↓      ↓   │
│ 选题工人  写作工人  修改工人  评分工人 发布工人│
│                                               │
│  飞书通知层：日常播报 + 异常告警 + 极少确认    │
└──────────────────────────────────────────────┘
```

**客户经理 Agent 核心能力**：
- **上下文记忆**：知道前天发过豆包文章，今天不再选豆包选题
- **质量判断**：选题分数不够高时不硬写，主动建议"今天不发"
- **异常兜底**：写作Agent出问题（风格不对、数据有误），自动调动修改Agent补救
- **智能分派**：根据选题类型自动选择最合适的写作模式和结构
- **节奏控制**：根据30天运营策略控制发布频率和内容类型配比

**人工职责**：战略方向调整、知识库维护、极端异常处理

## 各阶段对比总结

| 维度 | Phase 1 | Phase 2 | Phase 3 | Phase 4 | Phase 5 |
|------|:-------:|:-------:|:-------:|:-------:|:-------:|
| 选题 | AI推送 | AI评审 | AI决策 | AI决策 | AI决策 |
| 写作 | Claude | Claude | Claude | AI Worker | AI Worker |
| 修改 | 人工 | 人工 | 人工 | AI Worker | AI Worker |
| 评分 | 人工 | 人工 | 人工 | AI Worker | AI Worker |
| 发布 | 人工 | 人工 | 人工 | 人工确认 | AI自动 |
| 人工在链路中的位置 | 选题+改+发 | 选题+改+发 | 改+发 | 发 | 监督 |
| 飞书角色 | 选题面板 | 选题+AI评审 | 通知面板 | 终稿预览 | 播报+告警 |

## 不推翻重来的设计

每进入下一阶段，只是在 Coze 工作流中：
1. 加新的 Worker Agent 节点
2. 调某个审批节点的"人工/AI"模式
3. 补新的 Agent Prompt

不需要重建 Bot、不需要迁移数据、不需要改变层间接口（topic-brief.md）。
