# Agent/Worker 角色分类（终局展望）

最终形态下的 AI Agent 角色体系。

---

## 角色层级

```
                    ┌─────────────────┐
                    │  客户经理 Agent   │  ← Manager 层
                    │  (Account Mgr)   │
                    └────────┬────────┘
                             │
        ┌────────┬───────────┼───────────┬──────────┐
        ↓        ↓           ↓           ↓          ↓
   ┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐
   │选题工人│ │写作工人│ │修改工人│ │评分工人│ │发布工人│  ← Worker 层
   └────────┘ └────────┘ └────────┘ └────────┘ └────────┘
                             ↑
                    可被客户经理调度复用
```

---

## Manager 层

### 客户经理 Agent（Account Manager）

**定位**：公众号内容运营的"AI主编"，统筹从选题到发布的全流程。

**核心能力**：

| 能力 | 说明 |
|------|------|
| **上下文记忆** | 追踪已发布文章历史，避免选题重复、结构雷同、风格偏离 |
| **质量判断** | 选题不行时主动建议"不发"；文章不合格时退回重写 |
| **异常兜底** | 工人Agent出问题（风格偏离、事实错误），自动调动补救流程 |
| **节奏控制** | 根据30天运营策略控制发布频率、内容类型配比 |
| **向上汇报** | 每日/每周向飞书推送内容运营简报 |

**典型对话**：

> "老板，今天扫描到5个候选，3个≥12分。我选了豆包2.1那条，因为：
> 1. 上周没发过豆包相关内容
> 2. 评分最高（13/15），服务商痛点打得很准
> 3. 事实足够（官方发布+媒体解读+社群反馈），3类锚点都有
>
> 已经让写作工人出稿了，预计30分钟后给你看初稿。"

## Worker 层

### 1. 选题工人（Topic Scout）

| 属性 | 值 |
|------|-----|
| **职责** | 扫描7平台热点，按知识库规则评估打分，生成19字段选题Brief |
| **对应阶段** | 原 geo-content-writer Stage 0 |
| **输入** | 平台热点数据、关键词轮换表、历史选题记录 |
| **输出** | `topic-brief.md`（含三维评分+推荐理由） |
| **部署** | Coze Bot（Phase 1 即实现） |
| **使用的模型** | 便宜快速模型做扫描 + 强推理模型做评估 |

### 2. 写作工人（Draft Writer）

| 属性 | 值 |
|------|-----|
| **职责** | 根据Brief+风格卡+参考文章，用指定写作模式出初稿 |
| **对应阶段** | 原 geo-content-writer Stage 1-3 |
| **输入** | `topic-brief.md` + `style-analysis.md` + 参考文章 |
| **输出** | `article-draft.md`（含段落节奏标记、排版提示） |
| **部署** | Claude Code（初期）/ Coze Agent（后期） |
| **写作模式** | 教练模式 / 快速模式 / 混合模式 / 仿写模式 |

### 3. 修改工人（Humanizer / Editor）

| 属性 | 值 |
|------|-----|
| **职责** | 9项反AI检测清单检查 + 人味化改写 + 品牌声音校准 |
| **对应阶段** | 原 geo-content-writer Stage 4 |
| **输入** | `article-draft.md` + 品牌声音文档 |
| **输出** | `article-final.md`（通过人味化检查） |
| **核心检查** | 句长抖动 / 禁用词替换 / AI高频词黑名单 / 开头破冰 / 第一人称密度 / 事实密度 / 标点多样性 / 结构不完美 / 语气终审 |
| **部署** | Claude Code |

### 4. 评分工人（Quality Scorer）

| 属性 | 值 |
|------|-----|
| **职责** | 5维度自检打分 + 质量门禁 |
| **对应阶段** | 原 geo-content-writer Stage 4 的自检环节 + Stage 5 的发布前检查 |
| **输入** | `article-final.md` + 发布前检查清单 |
| **输出** | 评分报告（<35合格 / 35-50退回修改 / >50重写） |
| **部署** | Coze Agent（独立评分节点） |

**5维度评分标准**（从 geo-content-writer SKILL.md）：
1. 句长抖动（20%）— 短句占比 > 60%，最长/最短句比 > 5:1
2. AI高频词（20%）— 全文"系统性/全方位/进一步/深入"类词 < 3个
3. 事实密度（25%）— 每段至少1个具体事实（数字/日期/名称/政策）
4. 结构不完美（15%）— 小节长度不对称（最长 ≠ 最短）
5. 人味语气（20%）— 含口语化表达/自嘲/不确定性/个人观点

### 5. 发布工人（Publisher）

| 属性 | 值 |
|------|-----|
| **职责** | Markdown→微信HTML排版 + 配图需求生成 + 发布到草稿箱 |
| **对应阶段** | 原 geo-content-writer Stage 5-6 |
| **输入** | `article-final.md`（评分合格） |
| **输出** | 微信公众号草稿箱文章 + 发布后监测数据 |
| **部署** | Claude Code + wechat-publisher 工具 |

## 角色间协作关系

```
选题工人 ──(topic-brief.md)──→ 客户经理
                                    │
                         ┌──────────┘
                         ↓ (分配写作任务+指定模式)
写作工人 ──(article-draft.md)──→ 客户经理
                                    │
                         ┌──────────┘
                         ↓ (触发修改流程)
修改工人 ──(article-final.md)──→ 评分工人
                                    │
                         ┌──────────┘
                         ↓ (评分报告)
                        客户经理
                           │
              ┌────────────┼────────────┐
              ↓            ↓            ↓
           合格发布     退回修改     退回重写
              │         (修改工人)   (写作工人)
              ↓
          发布工人
              │
              ↓
        飞书"已发布"通知
              │
              ↓
        7天后数据复盘
```

所有角色之间通过标准Markdown文件交接，客户经理Agent统一调度和记录状态。
